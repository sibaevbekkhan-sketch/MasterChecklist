# AI OS: Coding Rules (CODING_RULES.md)
*Стандарты проектирования, качества, безопасности и расширяемости кода (Staff Engineer Layer). Гарантируют профессиональный уровень кодовой базы.*

---

## 1. Качество Кода и Читаемость (Clean Code Standards)
Код должен быть самодокументируемым, лаконичным и интуитивно понятным для любого разработчика.
* **Принцип единственной ответственности (Single Responsibility)**: Каждый модуль, класс и функция должны выполнять ровно одну задачу. Если функция занимает более 50 строк, рассмотреть целесообразность её декомпозиции.
* **Явное лучше неявного (Explicit is better than implicit)**: Избегать "магического" поведения, динамического изменения прототипов или скрытых побочных эффектов. Использовать строгую типизацию (например, TypeScript для JS, аннотации типов для Python).
* **Композиция против Наследования (Composition over Inheritance)**: Отдавать приоритет явной композиции, агрегированию и делегированию обязанностей через интерфейсы/протоколы перед сложным наследованием классов.

---

## 2. Безопасность Системы и Данных (Security by Design)
Безопасность — фундаментальное требование, которое никогда не приносится в жертву скорости реализации.
* **Нулевое доверие к вводу (Zero-Trust Input)**: Все внешние данные (параметры запросов, файлы, пользовательский ввод) должны проходить валидацию и санитарию (sanitizing) до начала их обработки. Защищать приложение от SQL/NoSQL-инъекций, XSS, CSRF и Path Traversal.
* **Защита секретов (Secrets Management)**: Никаких захардкоженных API-ключей, токенов, паролей или адресов серверов баз данных. Использовать механизм безопасного чтения из `.env` файлов или системного реестра/хранилища секретов.
* **Минимизация привилегий**: При написании системных скриптов использовать минимально необходимые системные привилегии, избегая выполнения кода от имени Администратора (root/Administrator), если это не критично.

---

## 3. Документирование и Архитектурная Трассировка
* **Техническое комментирование**: Комментировать не то, *что* делает код (это должно быть ясно из названий переменных и функций), а то, *почему* он это делает (бизнес-логика, нетривиальные технические решения, обходные пути для известных багов библиотек).
* **Спецификации интерфейсов**: Каждая публичная функция, класс или API-эндпоинт должны быть снабжены стандартными блоками документации (например, JSDoc для JavaScript/TypeScript, Docstrings для Python с описанием параметров, возвращаемых значений и выбрасываемых исключений).
* **Логирование**: Логирование должно быть структурированным, информативным и иметь уровни серьезности (`DEBUG`, `INFO`, `WARN`, `ERROR`). Запрещено логировать конфиденциальные персональные данные (PII) и учетные учетные записи.

---

## 4. Масштабируемость и Производительность
* **Оптимальные алгоритмы и структуры данных**: Выбирать структуры данных, соответствующие паттернам доступа к ним (например, использование хэш-таблиц для O(1) поиска). Избегать вложенных циклов там, где задачу можно решить за линейное время O(N).
* **Асинхронность и неблокирующий ввод-вывод**: Все операции ввода-вывода (работа с файловой системой, базой данных, внешними API) должны выполняться асинхронно, предотвращая блокировку основного потока выполнения (Event Loop / Main Thread).
* **Кэширование**: Повторяющиеся тяжелые вычисления или частые запросы к внешним API должны кэшироваться на соответствующем уровне (память, Redis, локальное хранилище) с обязательной стратегией инвалидации кэша.
* **Отказоустойчивость**: Любые сетевые вызовы должны содержать таймауты, механизмы повторных попыток (retries с экспоненциальной задержкой) и обработку сбоев через предохранители (Circuit Breakers) для предотвращения каскадных падений системы.
